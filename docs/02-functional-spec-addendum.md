# SmartNeighbor - שלב 2: הרחבת אפיון פונקציונלי

## מטרת המסמך

מסמך זה משלים את המפרט הפונקציונלי הראשי ומחדד מה נכנס ל-MVP, מהם תרחישי השימוש, ומהם הקריטריונים להצלחה.

## הגדרת MVP

מטרת ה-MVP היא להוכיח שאפשר לנהל חוויית דייר בסיסית דרך ממשק פשוט:

- דייר נכנס דרך Magic Link.
- רואה מאזן וחיובים אישיים.
- משלם דרך PayPal Sandbox או BIT simulator.
- מדווח תקלה דרך Agent.
- רואה Feed דמוי WhatsApp.
- ועד רואה Dashboard בסיסי.
- המערכת רצה ב-Docker וכוללת Kubernetes manifests.

## גבולות הדמו

| תחום | מה עובד בדמו | מה נדרש ל-production |
|---|---|---|
| PayPal | Sandbox order ו-capture | Live app, חשבון עסקי ואישורים |
| BIT | Simulator שמעדכן מאזן | TPP credentials, תעודות חתימה ו-Open Banking מלא |
| WhatsApp | Webhook scaffold ופיד פנימי | Meta Business App, מספר עסקי, webhook ציבורי |
| Agent | זיהוי כוונות rule-based | מודל AI מלא, memory, escalation |
| Database | PostgreSQL עם seed ו-migrations קלות | migration tool, backups, managed DB |
| Kubernetes | manifests מוכנים | cluster אמיתי, TLS, secrets, registry |

## User Stories

### דייר

- כדייר, אני רוצה לראות את החוב שלי בלי להיכנס עם סיסמה, כדי לשלם מהר.
- כדייר, אני רוצה לשלם ועד דרך כפתור ברור, כדי לא לבצע העברה ידנית.
- כדייר, אני רוצה לדעת על מה אני משלם, כדי להרגיש שקיפות.
- כדייר, אני רוצה לדווח תקלה בשפה חופשית, כדי לא למלא טופס.
- כדייר, אני רוצה להשאיל חפץ משכן, כדי לחסוך קנייה מיותרת.

### ועד בית

- כחבר ועד, אני רוצה לראות מי שילם ומי לא, בלי לפרסם זאת בקבוצה.
- כחבר ועד, אני רוצה לראות תקלות פתוחות ודחופות, כדי לטפל קודם במה שחשוב.
- כיו"ר ועד, אני רוצה שקיפות בהוצאות, כדי להסביר לדיירים לאן הכסף הולך.
- כחבר ועד, אני רוצה שכל הודעת WhatsApp חשובה תהפוך לנתון במערכת.

### ספק

- כספק, אני רוצה לקבל קריאת שירות ברורה עם תיאור, כתובת ודחיפות.
- כספק, אני רוצה לעדכן סטטוס עבודה בקלות.
- כספק, אני רוצה להעלות חשבונית בסיום עבודה.

## Acceptance Criteria

### חשבון דייר

- Magic Link מציג רק את נתוני הדייר הרלוונטי.
- דייר לא רואה חיובים של דייר אחר.
- מאזן מתעדכן אחרי תשלום.
- כאשר החוב הוא 0, כפתורי התשלום נחסמים ומופיע הסבר ברור.

### PayPal

- לחיצה על PayPal יוצרת order ב-Sandbox.
- המשתמש מועבר ל-approval URL של PayPal.
- לאחר חזרה, המערכת מבצעת capture ומעדכנת את המאזן.
- פרטי כרטיס בדיקה מוצגים ליד הכפתור, אך אינם נאספים באפליקציה.

### BIT

- לחיצה על BIT יוצרת תשלום simulator.
- התשלום מעדכן את ה-ledger ואת המאזן.
- אחרי תשלום מלא מופיע כפתור איפוס דמו לבדיקה נוספת.
- הקוד מבודד ב-adapter כדי לאפשר חיבור עתידי ל-API אמיתי.

### WhatsApp Feed

- הודעה שנשלחת בפיד המקומי מופיעה בצ'אט.
- רענון הודעות לא מקפיץ את המשתמש לראש הדף.
- webhook path קיים עבור Meta setup עתידי.

### Agent

- הודעת חירום מזוהה כ-emergency.
- שאלת תשלום מזוהה כ-payment query.
- בקשת חפץ מזוהה כ-borrow item.
- דיווח תקלה יוצר תגובת Agent מתאימה.

## דרישות לא פונקציונליות

### אבטחה

- אין שמירת פרטי כרטיס באפליקציה.
- פרטי סליקה נשארים אצל provider חיצוני.
- Committee actions עתידיות ידרשו 2FA.
- secrets נשמרים ב-.env או Kubernetes Secret.

### פרטיות

- Magic Link מוגבל לדייר יחיד.
- אין פרסום ציבורי של חייבים.
- WhatsApp processing דורש הסכמה מפורשת ב-production.
- דייר יוכל לבקש מחיקת מידע.

### ביצועים

- טעינת דמו מקומית עד 2 שניות בסביבה רגילה.
- API endpoints בסיסיים צריכים להגיב בפחות מ-500ms בדמו.
- frontend שומר על scroll בצ'אט בזמן polling.

### זמינות

- Docker Compose מתאים לדמו ופיתוח.
- Kubernetes manifests מאפשרים replicas ו-restart אוטומטי.
- production צריך managed database וגיבויים.

### נגישות ושימושיות

- ממשק RTL מלא.
- PWA במובייל.
- כפתורי פעולה ברורים.
- הודעות סטטוס אחרי תשלום או שגיאה.

## מסקנה

המפרט המקורי רחב ומתאים כחזון מוצר. ה-MVP הנוכחי מוכיח את הליבה: חשבון דייר, תשלום, Agent בסיסי, WhatsApp-first UI, Docker ו-Kubernetes. הדבר החשוב בהגשה הוא להציג בבירור מה מומש ומה נשאר ב-roadmap.
